# Operator UI — UX audit (2026-06-10)

Scope: code-only audit of `/mnt/d/code/business-os/packages/ui/src/**` plus the inbox-* agent manifests they consume. Every claim cites file:line. Opinion-first, no hedging.

---

## Top 5 things to fix first

1. **Agent detail page leads with Settings — wrong mental model.** `AgentDetail.tsx` builds a two-column grid where the literal first card the operator sees, in the top-left, is `Settings` (line 156-179). The second card is `Manual run input` (181-202), then `Connectors` (204-267), then `Recent runs` (269-328). The operator's job is "run this thing and see what happened" — Settings should be a gear, Connectors should be visible-but-quiet, and the page should open with **run state + last run summary + recent runs** dominating the layout. Today, the Run button is a small `btn-primary` shoved into the PageHeader's `right` slot (149-153), which is the smallest, least-noticed surface on the page. This single layout change probably fixes 60% of Matt's stated complaints.

2. **No way to add or remove agents from the UI — and the empty state admits it.** `AgentsList.tsx:48-52` literally tells the operator to go edit `business-os.config.ts` when there are zero agents. The whole "agent appears because it's in the registry" model surfaces here as a dead end. There's no Add Agent button anywhere, no per-row `…` menu, no delete, no disable. Matt wants "Add agent → pick from library → fill required settings inline → done." Until that lands, the agents page is a read-only list of things that magically appeared, which is exactly the backwards mental model he flagged.

3. **No Run-from-list and no inline status on AgentsList.** `AgentsList.tsx:64-87` renders three columns (Agent, Schedule, Last run). To trigger a manual run the operator must click into detail, scroll past Settings, find the small Run button in the header, then come back. For an operator-console-on-a-bunch-of-cron-jobs this is the single highest-frequency action and it's two clicks deep. Add a Run button to each row and a "running…" pill in the Last run column when a run is in-flight.

4. **RunDetail dumps raw JSON as the primary artifact.** `RunDetail.tsx:95-103` renders `JSON.stringify(run.details, null, 2)` inside a `<pre>` with `max-h-96 overflow-auto`. For the inbox-surface digest (which returns `{ digest: [{id, score, reason}, ...] }` per `agents/inbox-surface/src/index.ts:38-43`), this means the most operator-relevant page in the entire app is a wall of `\n`-escaped JSON. There's already CSV/Markdown export (161-186) — that proves the data is row-shaped. Default-render the most common shapes (array-of-objects → table) and keep the JSON as a "View raw" toggle.

5. **SchemaForm renders bare labels with zero help text because agent authors haven't called `.describe()`.** `agents/inbox-surface/src/index.ts:24-32` and `agents/inbox-categorize/src/index.ts:34-39` declare every setting with `z.number().int()...default(200)` and friends — no `.describe()` anywhere. `SchemaForm.tsx:213,238,262,288,309` only renders Help text when `schema.description` is set. So in production the operator sees labels like "Max Messages", "Window Days", "Vip Senders", "Confidence Threshold", "Unread Only" with no explanation of what they do or what good values look like. The fix is two-sided: (a) add `.describe()` to every settings field in every shipped agent (15 minutes of work), and (b) consider showing the default value inline ("default: 200") in the Help block so operators understand the units.

## Detail findings

### Shell + global navigation (`components/Shell.tsx`)

- **What's wrong:** The sidebar lumps Audit log and Settings under an "Operator" sublabel (line 67-69), but Dashboard / Agents / Connectors / Providers above it have no group label, which makes "Operator" look like a different kind of thing rather than a visual divider. The grouping is also semantically off: Providers is a sub-page of Connectors (it's the on/off marketplace for the same providers that appear in Connectors' Add dropdown) but they sit as peers.
- **Why it matters:** Operator has to learn two pages (Connectors + Providers) that are really one concept, and the sidebar visually mis-groups them.
- **Proposed fix:** In `Shell.tsx:43-70`, either (a) nest Providers under Connectors as a tab on `ConnectorsPage`, or (b) keep them as peers but add a "Setup" / "Operate" grouping with Providers under Setup. Smallest fix: add a "Providers" link as a tab on the Connectors page header and remove it from the sidebar.
- **Effort:** M

- **What's wrong:** `Shell.tsx:44` links to `/dashboard`, but `app.tsx:76` redirects `/` to `/dashboard` and login redirects to `/agents` (`Login.tsx:17`). So the post-login landing is Agents but the sidebar's first item is Dashboard. Mild inconsistency.
- **Why it matters:** Operator's mental "home" is ambiguous.
- **Proposed fix:** Pick one. For this product (operate-the-agents tool, not a metrics dashboard) Agents is the right home. Either redirect `/` → `/agents` and demote Dashboard, or land on `/dashboard` after login (`Login.tsx:17`).
- **Effort:** S

### Dashboard (`pages/Dashboard.tsx`)

- **What's wrong:** Tile 3 reads "Capabilities covered: 2 / 5" with sub-label "active providers" (54-59). This is developer jargon ("capabilities", "providers") for a tile whose answer is "your install is half-configured." The Capability coverage card (113-144) is more useful but uses the raw capability slug as the primary label (`{c.capability}` in font-mono — 122), e.g. the operator sees `file-storage`, `email-inbox`, `llm`. Those should be human labels.
- **Why it matters:** First impression of the app is mono-spaced jargon.
- **Proposed fix:** Maintain a small `capability → {label, blurb}` map (probably in `@business-os/connector-sdk` or a UI-local constant) and render `LLM`, `Email inbox`, `Email send`, `CRM`, `File storage`. Keep the slug as a `font-mono` aside if you want.
- **Effort:** S

- **What's wrong:** Recent-runs row uses `window.location.href = ...` for navigation (line 81) instead of `react-router`'s `useNavigate` / `<Link>`. Full page reload on every row click. Same pattern in `AgentDetail.tsx:291`.
- **Why it matters:** Loses the SPA's in-memory state, looks janky vs. the rest of the app, breaks back-button history.
- **Proposed fix:** Use `useNavigate()` and `navigate(\`/runs/\${r.id}\`)`. Or wrap the whole `<tr>` in a `<Link>` with `display: table-row` (or convert to a div-grid).
- **Effort:** S

### AgentsList (`pages/AgentsList.tsx`)

- **What's wrong:** No Run button per row (table at 64-87). No filter, no search, no status pill ("never run", "running now", "broken — last 3 failed"). The Schedule cell renders `manual` in muted gray (line 9) which is correct typography but offers no affordance — the operator can't act on it.
- **Why it matters:** Highest-frequency operator action is hidden behind a click-through.
- **Proposed fix:** Add a fourth column "Actions" containing a `btn-secondary` "Run" per row, calling the same `Api.runAgent(slug, {})` that `AgentDetail` uses. For agents whose `inputSchema` is non-trivial, open a small inline-modal instead of POSTing `{}`.
- **Effort:** M

- **What's wrong:** Empty state (48-52) instructs the operator to edit `business-os.config.ts`. That is the documented "register an agent" path today, but it's exactly the file-driven mental model Matt wants to invert.
- **Why it matters:** Empty state cements the wrong mental model.
- **Proposed fix:** Replace with "Add agent" CTA that opens a panel listing registered-but-not-installed agents (analogous to the Providers page for connectors). Until that exists, soften copy to "No agents installed yet — agents are registered in your client shell's config."
- **Effort:** L (UI is M, but the API + manifest registry probably need extending)

- **What's wrong:** The Schedule column shows `manual`, `cron 0 */6 * * *`, or `event: <topic>` (lines 6-10) but the operator can't change any of it. The agent SDK already declares schedule as the manifest's, but the system has no override.
- **Why it matters:** Matt explicitly flagged scheduling UI as missing. Today operators have to redeploy with a new manifest to change cadence.
- **Proposed fix:** Add a per-agent `scheduleOverride` field (DB-resident, mirrors manifest shape) and surface it on the AgentDetail page behind a "Schedule" subsection. Show effective schedule on AgentsList as `0 */6 * * *` with a small "edit" affordance. Out-of-scope for this audit but flag it.
- **Effort:** L (requires API + persistence)

### AgentDetail (`pages/AgentDetail.tsx`)

- **What's wrong:** Settings is the top-left card (line 156-179), making it the de facto primary action. The Run button is a tiny right-side button in `PageHeader` (149-153). Bindings — which gate whether the agent can run at all — are *below* Settings and only render conditionally (204).
- **Why it matters:** Hierarchy lies about what's important. Operator wants run / last result / recent runs front and center.
- **Proposed fix:** Reorder to a single-column layout (or hero + sidebar):
  - **Hero:** Last run summary card (status pill, started/ended, summary, "View details" link) + a large "Run now" button. If never run, hero is just the Run button + "Never run".
  - **Below hero:** Recent runs table (currently 269-328) — promote this.
  - **Sidebar / collapsed:** Bindings (always visible — they're what makes "Run" work), schedule (read-only for now), settings behind a gear icon at the top right of the header.
  - Manual run input collapses into a "Run with custom input" disclosure under the Run button, only relevant when `agent.inputSchema` exists.
- **Effort:** L (it's a real reflow, not a polish pass)

- **What's wrong:** Bindings section (204-267) calls capabilities by their raw slug in a `<span className="font-mono text-xs">` (228). The label dropdown options show `{inst.displayName} ({inst.providerSlug})` (248). Operator sees `email-inbox` next to `My Gmail (composio-gmail)`.
- **Why it matters:** Two layers of slug-jargon ("email-inbox", "composio-gmail") confront the operator at exactly the moment they're picking which account to wire up.
- **Proposed fix:** Use the human capability label (see Dashboard finding). For instance options, drop the parenthetical `providerSlug` — the displayName already includes "Gmail" or "Anthropic" from the auto-naming we just shipped.
- **Effort:** S

- **What's wrong:** "Manual run input" textarea is a JSON editor (190-200) for agents without an `inputSchema`. Inbox agents all declare `InputSchema = z.object({}).optional()` (`agents/inbox-surface/src/index.ts:22`), so this renders an empty `SchemaForm` with no fields — a section that takes up the entire top-right card to show nothing.
- **Why it matters:** Wastes the most valuable real estate on the page on a "{}" placeholder.
- **Proposed fix:** When `inputSchema` is empty or absent, suppress the Manual run input card entirely and surface the Run button alone. If/when a future agent has a non-empty input schema, re-show it.
- **Effort:** S

- **What's wrong:** No event-trigger awareness in the UI. Matt noted Gmail/Outlook via Composio is event-driven (webhooks) while IMAP is polled (cron) — both are bound to capability `email-inbox` — so the *effective* trigger depends on which connector instance is bound, not just the manifest. Currently `AgentDetail` shows nothing about how this agent fires.
- **Why it matters:** Operator can't tell whether the agent will run automatically when mail arrives or only on schedule. They'll bind IMAP and wonder why nothing fires until cron.
- **Proposed fix:** In the bindings table, after the operator picks an instance, show a small inline hint: "Triggers: webhook (instant)" for Composio-backed providers, "Triggers: polling every Xm" for IMAP. The connector instance metadata needs to expose this; add a `triggerMode: 'event' | 'poll' | 'manual'` to the provider definition and the bindings UI reads it.
- **Effort:** M

- **What's wrong:** Bindings auto-strip empty values on save (lines 62-65) but the saved-bindings success toast doesn't tell the operator a binding was *cleared* when they left a field blank.
- **Why it matters:** Mild — silent strip could surprise.
- **Proposed fix:** Toast could read "Saved — N bindings active, M cleared" when cleared > 0.
- **Effort:** S

### RunDetail (`pages/RunDetail.tsx`)

- **What's wrong:** Raw JSON dump (95-103) is the primary content. See Top 5 #4.
- **Why it matters:** Inbox digests are array-of-objects — perfect for a table.
- **Proposed fix:** Detect shape at render time:
  - `{ digest: [...] }` or `{ classifications: [...] }` etc → render each top-level array as a section with a table (one row per item, columns from the keys, a copy-row button per row).
  - For each row that has an `id` and the connector exposes deep links (Composio Gmail returns thread/message ids), render the id as a link to the connector's web UI.
  - Keep a "Show raw JSON" disclosure for everything.
  - The shape detection logic can live in `lib/export.ts` which already does similar work for CSV/Markdown (referenced at line 5).
- **Effort:** M

- **What's wrong:** `PageHeader` is being abused — `description` is typed as `string` (`PageHeader.tsx:5`) but `RunDetail.tsx:50-53` passes a `<span>` element cast through `as unknown as string`. The run id ends up as the page subtitle.
- **Why it matters:** Type lies, and the run id (a UUID) as the subtitle is operator-hostile — that subtitle should describe what they're looking at, not be a copy-pasteable id.
- **Proposed fix:** (a) Loosen `PageHeader.description` to `string | ReactNode`. (b) Set description to something like "How this run went and what it did" or just omit it. Put the UUID inline next to the title with a copy-to-clipboard button. Or move the UUID into the Outcome card as a "Run id" row.
- **Effort:** S

- **What's wrong:** Audit meta column renders `JSON.stringify(a.meta)` with no formatting (133) — same JSON-blob problem at smaller scale.
- **Why it matters:** Operator scanning a run's audit trail can't read meta at a glance.
- **Proposed fix:** Render meta as `key: value` pairs (one per line) when it's a flat object — the common case for `ctx.audit({ action: 'sent_email', meta: { to, subject } })`.
- **Effort:** S

- **What's wrong:** Back button is `← Agent` in the header right slot (54-58). Browser back works, but breadcrumbs would be more honest.
- **Why it matters:** Minor.
- **Proposed fix:** Optional — add a real breadcrumb (Agents / <displayName> / Run <id-short>) at the top.
- **Effort:** S

## Cross-cutting patterns

1. **Slug leakage everywhere.** `capability` (`email-inbox`, `llm`, `file-storage`), `providerSlug` (`composio-gmail`, `anthropic`), `authKind` (`api-key`, `oauth2`, `custom`), `agent.slug` (`inbox-surface`) all appear as user-facing labels in font-mono. They're convenient for grepping audit logs but they say "this is a developer tool, not a polished product." Pick a rule: slugs are *secondary* metadata, always — never the heading. Build a `humanizeCapability(slug)` / `humanizeProvider(slug)` once, use everywhere. Files affected: `Dashboard.tsx:122`, `AgentDetail.tsx:228, 248`, `ConnectorsPage.tsx:145, 531`, `ProvidersPage.tsx:99, 121`.

2. **No empty-state CTAs.** Every empty state today is a paragraph of explanation (`AgentsList.tsx:48`, `Dashboard.tsx:64`, `AgentDetail.tsx:272`, `AuditPage.tsx:109`, `ConnectorsPage`'s missing-providers branch at 159-163, `RunDetail.tsx:100, 109`). None point to a button. For a high-touch operator console, empty states should be "Here's nothing yet, [click this to make something]." Especially `AgentsList`, `ConnectorsPage` (when zero instances), and `AgentDetail` (when zero bindings: "Add a Gmail account →" jumps the operator to /connectors with the right capability pre-opened).

3. **Hierarchy by card grid instead of by importance.** AgentDetail and RunDetail both use `grid lg:grid-cols-2` putting cards in reading order regardless of priority. The operator's eye reads top-left first, but the most important content (Run button on AgentDetail, run outcome on RunDetail) isn't always there. Reflow these to hero + supporting cards.

4. **State sync after writes is full reloads.** `AgentDetail.reload()` (38-55) refetches agent, runs, and connectors after every save/run. `ConnectorsPage.reload()` does the whole connector list. That's fine while data is small but it's a noticeable flash on slower connections. Pattern: optimistic local updates with a quieter background reconcile, like `ProvidersPage.toggle` (53-67) already does well.

5. **Inconsistent navigation primitives.** Some click handlers use `<Link>` (good — `Dashboard:88`, `AgentDetail:208`), others use `window.location.href` (bad — `Dashboard:81`, `AgentDetail:291`). Pick one. The `Link`-wrapped rows in tables need conversion to grid or `display: contents` to keep table semantics.

6. **Error treatment is consistent and decent.** Same red card pattern (`border-red-200 bg-red-50`...) appears across every page. The `apiErrorMessage` helper (`AgentDetail.tsx:10-19`, `ConnectorsPage.tsx:8-26`) is duplicated in both files — promote to `lib/api-errors.ts`.

7. **Toast + inline error doubled up.** `ConnectorsPage.addInstance` (60-74) deliberately shows both a toast and an inline error block. Good — comment at 68-70 explains why. But not every page does this; AgentDetail's `saveSettings` (94-105) only toasts. Pick a rule: form errors inline, action errors toast.

8. **No loading skeletons.** `!data ? "Loading…"` (Dashboard:36, AgentsList:47, ConnectorsPage:137, ProvidersPage:91, AuditPage:107). On every navigation the operator sees a flash of "Loading…" before the page paints. Skeleton states (gray rectangles where data will be) would feel faster.

9. **Forms have no save-on-Enter / no dirty tracking.** AgentDetail's settings form has a "Save settings" button (175-177) but no warning if you navigate away with unsaved changes, and Enter inside the form does nothing useful. Bindings same problem. Acceptable for v1 but consider `beforeunload` + a dirty flag.

10. **No live updates.** Once you fire a run, you have to manually reload AgentDetail or RunDetail to see it complete. `AgentDetail.runNow` (107-127) does a `setTimeout(reload, 500)` once. SSE or polling on RunDetail until `endedAt` is set would close the loop nicely. (Not a blocker.)

## Open questions for Matt

1. **Add Agent UX direction.** When you say "Add agent → configure required settings inline → done" — is the source list "every agent package available on the registry" (server queries the registered-agents list, including disabled ones) or "every agent registered in `business-os.config.ts` that isn't yet enabled"? I.e. is enabling fully data-driven from the DB like Providers, or do agents stay file-declared with a DB on/off bit?

2. **Settings-behind-a-gear interaction.** When the operator clicks the gear on a configured agent, do you want a slide-over drawer (Radix Dialog), a route push (`/agents/:slug/settings`), or an inline expand on the same page? Drawer feels right for "set once, occasional tweaks" but it's more code.

3. **Run-from-list for agents with non-trivial input schemas.** Today all inbox agents take `{}`. When an agent does need real input (e.g. a future "send-followup" agent with a template), what's the desired AgentsList Run experience? Disable the row button and force click-through to AgentDetail? Or pop a small input modal?

4. **Schedule UI scope.** Just operator override for cron expression, or full schedule editor including switching kind (manual ↔ cron)? The latter implies the agent needs to support being invoked either way, which most do, but some event-driven agents may not have a manual entry point.

5. **Event-vs-poll display.** Where do you want the "this binding fires on webhook" / "this binding polls every 5m" indicator — on the AgentDetail bindings row, on the ConnectorsPage instance card, both? My take: AgentDetail bindings only (the operator cares about effective trigger when configuring the agent, not when configuring the connector).

6. **Capability + provider humanization source of truth.** Should the human labels live in `@business-os/connector-sdk` (provider authors set their own `displayName` already — extend with `capabilityLabel`?), in `@business-os/ui` as a static map, or returned from the server with each capability? Server-returned is most flexible but most code.

7. **Acronym handling in SchemaForm labels.** Add a hardcoded acronym list in `humanLabel` (cheap, brittle) or change agent authors to use `.describe('VIP senders | ...')` and treat anything before `|` as a label override (more work per field, more consistent)?

8. **/providers vs /connectors merge.** Worth a flag day to collapse into one page with tabs, or keep separate? My recommendation: tabs on /connectors, remove /providers from sidebar.

9. **RunDetail rendering strategy.** Should "schema-aware" mean (a) the agent declares an `outputSchema` (Zod) and the framework auto-renders that, or (b) the framework just sniffs for array-of-objects shapes and renders tables?
